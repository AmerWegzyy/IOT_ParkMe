# ParkMe — Firestore Database Structure

## Overview

ParkMe uses **Google Firebase Firestore** as its database. Firestore is a **NoSQL document database** — instead of SQL tables with rows and columns, data is organized as **collections** containing **documents** (JSON objects).

---

## Collections

The database has **4 collections**, each corresponding to an old SQL table:

```
🔥 Firestore Database
│
├── 📁 users/              → Registered users (students, lecturers, staff, admins)
├── 📁 vehicles/            → Vehicles linked to users by license plate
├── 📁 parking_spots/       → Live state of each physical parking sensor (ESP32)
└── 📁 parking_logs/        → Historical record of every parking event
```

---

## Collection Details

### 1. `users` Collection

Stores all registered users of the system.

| Field | Type | Description |
|-------|------|-------------|
| `name` | string | Full name |
| `email` | string | Email address (used for login) |
| `role` | string | One of: `student`, `lecturer`, `staff`, `special-needs-driver`, `admin` |
| `points` | number | Reward points |
| `created_at` | timestamp | Account creation time |

- **Document ID:** Auto-generated by Firestore
- **Queried by:** `email` field (for login)

**Example document** (`users/abc123`):
```json
{
  "name": "John Doe",
  "email": "student@technion.ac.il",
  "role": "student",
  "points": 10,
  "created_at": "2026-06-14T12:00:00Z"
}
```

---

### 2. `vehicles` Collection

Maps license plates to their owners.

| Field | Type | Description |
|-------|------|-------------|
| `user_id` | string | Reference to the owner's document ID in `users/` |
| `created_at` | timestamp | Registration time |

- **Document ID:** The license plate number (e.g., `"1234567"`)
- **Queried by:** Direct document lookup by license plate

**Example document** (`vehicles/1234567`):
```json
{
  "user_id": "abc123",
  "created_at": "2026-06-14T12:00:00Z"
}
```

> To find a vehicle's owner: read `vehicles/{plate}` → get `user_id` → read `users/{user_id}`

---

### 3. `parking_spots` Collection

Represents the **live state** of each physical ESP32 sensor node. Updated on every heartbeat from the hardware.

| Field | Type | Description |
|-------|------|-------------|
| `mac_address` | string | Unique MAC address of the ESP32 node |
| `category` | string | Who can park here: `student`, `lecturer`, `staff`, `special-needs-driver` |
| `is_occupied` | boolean | Is a car physically detected by the sensor? |
| `battery_level` | number | Battery percentage (0–100) |
| `last_seen` | timestamp | Last heartbeat received from this sensor |

- **Document ID:** The spot label (e.g., `"A1"`, `"B2"`)
- **Queried by:** Direct document lookup by spot ID, or by `mac_address` field (for heartbeats)

**Example document** (`parking_spots/A1`):
```json
{
  "mac_address": "AA:BB:CC:DD:EE:01",
  "category": "student",
  "is_occupied": true,
  "battery_level": 95.5,
  "last_seen": "2026-06-14T15:30:00+03:00"
}
```

---

### 4. `parking_logs` Collection

Records every parking event — entry, exit, violations. This is the **history** of the system.

| Field | Type | Description |
|-------|------|-------------|
| `spot_id` | string | Which parking spot (e.g., `"A1"`) |
| `license_plate` | string | Plate number, or `"UNIDENTIFIED"` / `"ABORTED"` / `"RESOLVED"` |
| `user_id` | string or null | Owner's document ID in `users/`, null if unknown |
| `snapshot_role` | string or null | User's role at the time of parking |
| `entry_time` | timestamp | When the car arrived |
| `exit_time` | timestamp or null | When the car left (`null` = still parked) |
| `is_violation` | boolean | Was this an unauthorized parking event? |

- **Document ID:** Auto-generated by Firestore
- **Queried by:** `spot_id` + `exit_time == null` (to find active sessions), `is_violation == true` (for admin logs)

**Example document** (`parking_logs/xyz789`):
```json
{
  "spot_id": "A1",
  "license_plate": "1234567",
  "user_id": "abc123",
  "snapshot_role": "student",
  "entry_time": "2026-06-14T14:00:00+03:00",
  "exit_time": null,
  "is_violation": false
}
```

## How Relationships Work (No Foreign Keys!)

In SQL, relationships were **enforced by the database** using foreign keys. In Firestore, relationships are just **strings pointing to other documents** — the code is responsible for following them.

```
┌──────────────┐       user_id        ┌──────────────┐
│   vehicles/  │ ──── "abc123" ────→  │   users/     │
│   1234567    │                      │   abc123     │
└──────────────┘                      └──────────────┘

┌──────────────┐       spot_id        ┌──────────────┐
│ parking_logs/│ ──── "A1" ────────→  │parking_spots/│
│   xyz789     │                      │     A1       │
│              │       user_id        │              │
│              │ ──── "abc123" ────→  users/abc123   │
└──────────────┘                      └──────────────┘
```

### Example: "Who parked in spot A1?"

In SQL (one query):
```sql
SELECT u.name, v.license_plate
FROM parking_logs l
JOIN vehicles v ON l.license_plate = v.license_plate
JOIN users u ON v.user_id = u.id
WHERE l.spot_id = 'A1' AND l.exit_time IS NULL;
```

In Firestore (multiple steps in Python):
```python
# Step 1: Find active log for spot A1
logs = db.collection("parking_logs")
    .where("spot_id", "==", "A1")
    .where("exit_time", "==", None)
    .get()

# Step 2: Get the license plate from the log
plate = logs[0].to_dict()["license_plate"]

# Step 3: Look up the vehicle
vehicle = db.collection("vehicles").document(plate).get()

# Step 4: Look up the user
user = db.collection("users").document(vehicle.to_dict()["user_id"]).get()
```

---

## Special Values in `license_plate`

| Value | Meaning |
|-------|---------|
| `"1234567"` (digits) | Normal — a real license plate was read by OCR |
| `"UNIDENTIFIED"` | Sensor detected a car but the camera failed to read the plate |
| `"ABORTED"` | Car left within 60 seconds (bouncing driver — not a real park) |
| `"RESOLVED"` | Admin manually resolved an UNIDENTIFIED anomaly |

## Firestore Indexes Required

Firestore requires **composite indexes** for queries that filter/sort on multiple fields. The following queries in the code will need indexes (Firestore will auto-prompt you to create them on first use):

| Query | Fields | Used In |
|-------|--------|---------|
| Active logs for a spot | `spot_id` + `exit_time` + `entry_time` (desc) | Heartbeat endpoint |
| Violation logs | `is_violation` + `entry_time` (desc) | Admin logs endpoint |
| Unidentified logs | `license_plate` + `entry_time` (desc) | Admin logs endpoint |
| Resolve query | `spot_id` + `license_plate` + `exit_time` | Resolve endpoint |

> **Tip:** When you first call these endpoints, Firestore will return an error with a direct link to create the required index. Just click the link and it's done.
